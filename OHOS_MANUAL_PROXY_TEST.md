# system_proxy OHOS 手动代理测试说明

## 1. 测试目标

验证 system_proxy 在 OpenHarmony/HarmonyOS 设备上，是否能够正确读取系统中已配置的手动代理信息。

当前仓库中的 OHOS 插件实现位于 `ohos/src/main/ets/components/plugin/SystemProxyPlugin.ets`，核心逻辑是通过 `@ohos.net.connection.getDefaultHttpProxy()` 读取系统代理；示例工程位于 `example/`，页面启动后会调用 `SystemProxy.getProxySettings()`，并在界面上展示代理结果。

## 2. 适用场景

- 场景 A：设备未设置代理，验证是否显示 `no proxy`
- 场景 B：设备设置了手动代理，验证是否显示 `host:port`
- 场景 C：设备设置了手动代理，但界面仍显示异常，定位是系统设置问题还是插件读取问题

## 3. 前置条件

执行测试前，需要满足以下条件：

1. 已在电脑上编译并安装示例工程到 OHOS 设备。
2. 设备与电脑处于可正常联网状态。
3. 示例工程启动后能进入首页，并看到类似如下文本：

   `proxy setting on: ...`

4. 如果要验证“有代理”场景，需要提前准备一个可填写的代理地址，例如：

   - 代理主机：电脑局域网 IP
   - 代理端口：代理工具监听端口，例如 `8888`

## 4. OHOS 手动代理设置步骤

以下步骤用于在设备上配置手动代理：

1. 打开设备的 `设置`。
2. 进入 `WLAN` 或 `Wi-Fi`。
3. 选择当前连接的无线网络。
4. 打开网络详情或编辑网络配置页面。
5. 找到 `代理` 相关配置项。
6. 将代理模式切换为 `手动`。
7. 输入代理主机和端口。

示例：

- 主机：`10.50.80.71`
- 端口：`8888`

8. 保存配置。
9. 确认设备仍保持在该 Wi-Fi 下联网。

## 5. 示例工程验证步骤

### 5.1 无代理场景

1. 确认设备的 Wi-Fi 代理设置为关闭。
2. 启动示例应用。
3. 观察首页文本。

预期结果：

- 页面显示：`proxy setting on: no proxy`

### 5.2 手动代理场景

1. 按前文步骤在设备中配置手动代理。
2. 关闭示例应用后重新打开。
3. 观察首页文本。

预期结果：

- 页面显示：`proxy setting on: 代理主机:代理端口`
- 例如：`proxy setting on: 10.50.80.71:8888`

说明：

- 示例工程当前只展示 `host:port` 文本。
- 如果代理读取成功，页面会直接显示配置值。

## 6. 结果判定

### 6.1 符合预期

满足以下任一条件，可认为功能正常：

- 未设置代理时，显示 `no proxy`
- 设置手动代理后，显示正确的 `host:port`

### 6.2 不符合预期

如出现以下情况，需要进一步排查：

1. 页面一直停留在 `loading`
   - 说明 Dart 调用未完成，可能是原生插件没有正常回包。

2. 页面显示 `Failed to get system proxy.`
   - 说明平台侧抛出了异常并返回给了 Dart。

3. 已设置手动代理，但页面仍显示 `no proxy`
   - 说明系统层没有把代理信息返回给插件，或当前网络下代理未真正生效。

4. 页面显示的 `host:port` 与设备设置不一致
   - 说明读取结果异常，需要检查设备系统代理配置是否已经被其他策略覆盖。

## 7. 建议排查顺序

如果结果不符合预期，建议按以下顺序排查：

1. 先检查设备系统里代理是否真的保存成功。
2. 再重启示例应用，避免页面停留在旧状态。
3. 查看 OHOS 侧日志，确认是否进入 `SystemProxyPlugin` 的 `getProxySettings`。
4. 如果日志中出现 `System internal error`，优先判断是否为系统接口 `getDefaultHttpProxy()` 返回异常。
5. 如果无日志输出，优先检查 MethodChannel 名称是否一致，当前应为 `system_proxy`。

## 8. 结论示例

示例结论可写为：

`在 OHOS 设备上完成手动代理配置后，启动 example 应用，页面正确显示已设置的 host:port，说明 system_proxy 的 OHOS 手动代理读取能力符合预期。`
